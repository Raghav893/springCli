     9тЖТ     9тЖТ
    10тЖТ    10тЖТ## 0. One-line pitch
    11тЖТ    11тЖТ
    12тЖТ    12тЖТ`springup` is what `npm create`, `bun create`, and `cargo generate` are to their ecosystems,
    13тЖТ    13тЖТbut for Spring Boot: a fast, single static binary that interactively (or non-interactively)
    14тЖТ    14тЖТscaffolds a production-ready Spring Boot backend by combining the official Spring Initializr
    15тЖТ    15тЖТAPI with an opinionated custom template layer (architecture skeletons, Docker, CI, config
    16тЖТ    16тЖТprofiles) тАФ written in Rust, distributed as a real CLI tool.
    17тЖТ    17тЖТ
    18тЖТ    18тЖТ---
    19тЖТ    19тЖТ
    20тЖТ    20тЖТ## 1. Goals & non-goals
    21тЖТ    21тЖТ
    22тЖТ    22тЖТ**Goals**
    23тЖТ    23тЖТ- Feel as polished and fast as `bun create` / `create-t3-app`: sub-second startup, smooth
    24тЖТ    24тЖТ  interactive prompts, sensible defaults, zero Java/Maven/Gradle required to just *generate*
    25тЖТ    25тЖТ  a project.
    26тЖТ    26тЖТ- Two first-class modes: interactive TUI wizard (default) and fully flag-driven non-interactive
    27тЖТ    27тЖТ  mode (for CI, scripting, muscle-memory power users).
    28тЖТ    28тЖТ- Generate real, buildable Spring Boot projects by delegating dependency/version resolution to
    29тЖТ    29тЖТ  the official Spring Initializr API (`https://start.spring.io`), then layering custom
    30тЖТ    30тЖТ  extras on top (architecture skeleton, Dockerfile, GitHub Actions, application.yml profiles).
    31тЖТ    31тЖТ- Be genuinely production-grade as a piece of software: proper error handling, logging,
    32тЖТ    32тЖТ  tests, CI, versioned releases, cross-platform binaries, shell completions, self-update.
    33тЖТ    33тЖТ- Support an "add" workflow later (`springup add module`) тАФ so today's design must not paint
    34тЖТ    34тЖТ  the tool into a one-shot-generator corner. Use a lightweight project config file
    35тЖТ    35тЖТ  (`springup.toml`) written into generated projects to enable this later without a rewrite.
    36тЖТ    36тЖТ
    37тЖТ    37тЖТ**Non-goals (explicitly out of scope for v1)**
    38тЖТ    38тЖТ- No frontend/monorepo scaffolding (no Next.js/React generation). Backend + Docker/CI only.
    39тЖТ    39тЖТ- No bundling of a JVM, Maven, or Gradle. The tool never requires Java to be installed to
    40тЖТ    40тЖТ  *generate* a project (it may shell out to `./mvnw`/`./gradlew` only for optional post-gen
    41тЖТ    41тЖТ  steps like "run tests now?", which are opt-in and gracefully skipped if unavailable).
    42тЖТ    42тЖТ- No GUI. Terminal only.
    43тЖТ    43тЖТ- No plugin marketplace/registry in v1 (design for it, don't build it yet).
    44тЖТ    44тЖТ
    45тЖТ    45тЖТ---
    46тЖТ    46тЖТ
    47тЖТ    47тЖТ## 2. Prior art to study before writing code
    48тЖТ    48тЖТ
    49тЖТ    49тЖТThe agent should mentally (or actually, if it has web access) benchmark against:
    50тЖТ    50тЖТ- `cargo-generate` (Rust) тАФ template mechanics, `{{placeholder}}` substitution patterns.
    51тЖТ    51тЖТ- `create-t3-app` (TS) тАФ interactive prompt UX, layered "optional add-on" selection.
    52тЖТ    52тЖТ- `starship` (Rust) тАФ how a beloved Rust CLI handles config file precedence, cross-platform
    53тЖТ    53тЖТ  installs, and shell completions.
    54тЖТ    54тЖТ- `bun create` / `npm create` тАФ speed expectations and flag ergonomics.
    55тЖТ    55тЖТ- Spring Initializr itself (`start.spring.io`) тАФ this tool is a client of its REST API, not a
    56тЖТ    56тЖТ  replacement for it. Read its API docs (`https://start.spring.io` returns machine-readable
    57тЖТ    57тЖТ  metadata at the root when queried with `Accept: application/json`).

    61тЖТ    61тЖТ## 3. Core architecture
    62тЖТ    62тЖТ
    63тЖТ    63тЖТ### 3.1 High-level flow
    64тЖТ    64тЖТ
    65тЖТ    65тЖТ```
    66тЖТ    66тЖТUser runs `springup new`
    67тЖТ    67тЖТ        тФВ
    68тЖТ    68тЖТ        тЦ╝
    69тЖТ    69тЖТ[Mode detection] тФАтФА flags present & non-interactive? тФАтФАтЦ║ skip wizard, validate flags
    70тЖТ    70тЖТ        тФВ no
    71тЖТ    71тЖТ        тЦ╝
    72тЖТ    72тЖТ[Interactive Wizard] (ratatui or dialoguer-based prompts)
    73тЖТ    73тЖТ   1. Project metadata (group, artifact, name, package, description)
    74тЖТ    74тЖТ   2. Spring Boot version (fetched live from Initializr metadata, cached)
    75тЖТ    75тЖТ   3. Build tool: Maven | Gradle (Groovy) | Gradle (Kotlin)
    76тЖТ    76тЖТ   4. Language: Java | Kotlin
    77тЖТ    77тЖТ   5. Java version
    78тЖТ    78тЖТ   6. Packaging: Jar | War
    79тЖТ    79тЖТ   7. Dependencies (multi-select, searchable, grouped by category тАФ from Initializr metadata)
    80тЖТ    80тЖТ   8. [Custom layer] Architecture skeleton? (none / layered / hexagonal) тАФ optional, user choice
    81тЖТ    81тЖТ   9. [Custom layer] Extras multi-select: Dockerfile, docker-compose.yml, GitHub Actions CI,
    82тЖТ    82тЖТ      application-{dev,prod}.yml profiles, .editorconfig, pre-commit hook, README badges
    83тЖТ    83тЖТ        тФВ
    84тЖТ    84тЖТ        тЦ╝
    85тЖТ    85тЖТ[Resolve & Plan] тАФ build a ProjectPlan struct (see ┬з3.3), validate combos
    86тЖТ    86тЖТ        тФВ
    87тЖТ    87тЖТ        тЦ╝
    88тЖТ    88тЖТ[Fetch base project from Initializr] (zip stream) тФАтФАтЦ║ extract to temp dir
    89тЖТ    89тЖТ        тФВ
    90тЖТ    90тЖТ        тЦ╝
    91тЖТ    91тЖТ[Apply custom template layer] тАФ inject architecture folders, Docker/CI files,
    92тЖТ    92тЖТ   rewrite application.yml, write springup.toml manifest
    93тЖТ    93тЖТ        тФВ
    94тЖТ    94тЖТ        тЦ╝
    95тЖТ    95тЖТ[Post-gen] тАФ git init (optional), initial commit (optional), print summary + next steps
    96тЖТ    96тЖТ```
    97тЖТ    97тЖТ
    98тЖТ    98тЖТ### 3.2 Crate/module layout
    99тЖТ    99тЖТ
   100тЖТ   100тЖТ```
   101тЖТ   101тЖТspringup/
   102тЖТ   102тЖТтФЬтФАтФА Cargo.toml
   103тЖТ   103тЖТтФЬтФАтФА crates/
   104тЖТ   104тЖТтФВ   тФЬтФАтФА springup-cli/          # binary crate тАФ arg parsing, wizard orchestration, output
   105тЖТ   105тЖТтФВ   тФВ   тФЬтФАтФА src/main.rs
   106тЖТ   106тЖТтФВ   тФВ   тФЬтФАтФА src/cli.rs         # clap definitions
   107тЖТ   107тЖТтФВ   тФВ   тФЬтФАтФА src/wizard/        # interactive prompt flows
   108тЖТ   108тЖТтФВ   тФВ   тФЬтФАтФА src/commands/      # new.rs, add.rs, config.rs, completions.rs, update.rs
   109тЖТ   109тЖТтФВ   тФВ   тФФтФАтФА src/ui/            # terminal rendering helpers, spinners, theming
   110тЖТ   110тЖТтФВ   тФЬтФАтФА springup-core/         # library crate тАФ pure logic, fully unit-testable, no I/O-heavy CLI concerns
   111тЖТ   111тЖТтФВ   тФВ   тФЬтФАтФА src/plan.rs        # ProjectPlan struct + validation
   112тЖТ   112тЖТтФВ   тФВ   тФЬтФАтФА src/initializr/    # Spring Initializr API client + response models
   113тЖТ   113тЖТтФВ   тФВ   тФЬтФАтФА src/template/      # custom template engine (rendering, placeholder substitution)
   114тЖТ   114тЖТтФВ   тФВ   тФЬтФАтФА src/manifest.rs    # springup.toml read/write (project-level state)
   115тЖТ   115тЖТтФВ   тФВ   тФФтФАтФА src/config.rs      # global user config (~/.config/springup/config.toml)
   116тЖТ   116тЖТтФВ   тФФтФАтФА springup-templates/    # embedded custom templates (architecture skeletons, CI, Docker)
   117тЖТ   117тЖТтФВ       тФФтФАтФА assets/
   118тЖТ   118тЖТтФВ           тФЬтФАтФА architectures/layered/
   119тЖТ   119тЖТтФВ           тФЬтФАтФА architectures/hexagonal/
   120тЖТ   120тЖТтФВ           тФЬтФАтФА docker/
   121тЖТ   121тЖТтФВ           тФЬтФАтФА ci/github-actions/
   122тЖТ   122тЖТтФВ           тФФтФАтФА config-profiles/
   123тЖТ   123тЖТтФЬтФАтФА xtask/                     # cargo-xtask pattern: release automation, template asset checks
   124тЖТ   124тЖТтФФтФАтФА tests/                     # integration tests (see ┬з8)

   126тЖТ   126тЖТ
   127тЖТ   127тЖТRationale for splitting `springup-core` out: it must be usable without a terminal (unit
   128тЖТ   128тЖТtests, potential future library consumers, WASM playground, etc.) and keeps `main.rs` thin.
   129тЖТ   129тЖТ
   130тЖТ   130тЖТ### 3.3 Core data model
   131тЖТ   131тЖТ
   132тЖТ   132тЖТ```rust
   133тЖТ   133тЖТ/// The fully-resolved, validated description of what to generate.
   134тЖТ   134тЖТ/// Built by EITHER the wizard OR flag parsing тАФ both paths converge here.
   135тЖТ   135тЖТpub struct ProjectPlan {
   136тЖТ   136тЖТ    pub group_id: String,
   137тЖТ   137тЖТ    pub artifact_id: String,
   138тЖТ   138тЖТ    pub name: String,
   139тЖТ   139тЖТ    pub description: String,
   140тЖТ   140тЖТ    pub package_name: String,
   141тЖТ   141тЖТ    pub spring_boot_version: String,
   142тЖТ   142тЖТ    pub build_tool: BuildTool,       // Maven | GradleGroovy | GradleKotlin
   143тЖТ   143тЖТ    pub language: Language,          // Java | Kotlin
   144тЖТ   144тЖТ    pub java_version: String,
   145тЖТ   145тЖТ    pub packaging: Packaging,        // Jar | War
   146тЖТ   146тЖТ    pub dependencies: Vec<DependencyId>,   // Initializr dependency ids, e.g. "web", "data-jpa"
   147тЖТ   147тЖТ    pub architecture: Option<ArchitectureKind>, // None | Layered | Hexagonal
   148тЖТ   148тЖТ    pub extras: Vec<ExtraFeature>,   // Docker, DockerCompose, GithubActionsCi, ConfigProfiles, ...
   149тЖТ   149тЖТ    pub output_dir: PathBuf,
   150тЖТ   150тЖТ    pub git_init: bool,
   151тЖТ   151тЖТ    pub initial_commit: bool,
   152тЖТ   152тЖТ}
   153тЖТ   153тЖТ```
   154тЖТ   154тЖТ
   155тЖТ   155тЖТKeep this struct serializable (`serde`). It doubles as the schema for:
   156тЖТ   156тЖТ- Non-interactive flags (`clap` derives directly into a builder for this struct).
   157тЖТ   157тЖТ- A future `--from-file plan.toml` mode (batch/reproducible generation тАФ trivial to add
   158тЖТ   158тЖТ  later since the plan is already data, not scattered logic).
   159тЖТ   159тЖТ
   160тЖТ   160тЖТ### 3.4 Spring Initializr integration
   161тЖТ   161тЖТ
   162тЖТ   162тЖТ- Base URL: `https://start.spring.io`.
   163тЖТ   163тЖТ- On startup (or on a `springup update-metadata` command / cached with TTL), fetch
   164тЖТ   164тЖТ  `GET https://start.spring.io` with `Accept: application/json` to get the full metadata
   165тЖТ   165тЖТ  document: available Spring Boot versions, dependency list with categories/descriptions/
   166тЖТ   166тЖТ  compatibility ranges, Java versions, packaging options, language options.
   167тЖТ   167тЖТ- Cache this metadata locally (`~/.cache/springup/initializr-metadata.json` or platform
   168тЖТ   168тЖТ  equivalent via the `directories` crate) with a TTL (e.g. 24h) so the wizard is instant on
   169тЖТ   169тЖТ  repeat runs and works offline using stale-but-usable cached data (warn the user it's stale).
   170тЖТ   170тЖТ- To generate the actual project, call
   171тЖТ   171тЖТ  `GET https://start.spring.io/starter.zip?type=...&language=...&bootVersion=...&groupId=...&artifactId=...&name=...&packageName=...&packaging=...&javaVersion=...&dependencies=...`
   172тЖТ   172тЖТ  and stream the zip response to a temp file, then extract.
   173тЖТ   173тЖТ- Handle Initializr downtime/errors gracefully: clear error message, suggest retry, never
   174тЖТ   174тЖТ  panic. Consider a `--offline` flag that only works if a previously-downloaded base template
   175тЖТ   175тЖТ  cache exists (stretch goal, not required for v1).
   176тЖТ   176тЖТ- Validate dependency IDs against fetched metadata *before* hitting the generate endpoint, so
   177тЖТ   177тЖТ  users get a fast, friendly "unknown dependency 'foo', did you mean 'flyway'?" instead of a
   178тЖТ   178тЖТ  raw API error. Implement fuzzy matching (`strsim` crate, e.g. Jaro-Winkler) for suggestions.
   179тЖТ   179тЖТ
   180тЖТ   180тЖТ### 3.5 Custom template layer (the "extras" system)
   181тЖТ   181тЖТ
   182тЖТ   182тЖТThis is what differentiates `springup` from a bare Initializr CLI wrapper. Runs *after* the
   183тЖТ   183тЖТbase project is extracted.
   184тЖТ   184тЖТ
   185тЖТ   185тЖТ**Architecture skeletons** (optional, chosen interactively per ┬з3.1 step 8):
   186тЖТ   186тЖТ- `layered`: generates `controller/`, `service/`, `repository/`, `dto/`, `entity/`,
   187тЖТ   187тЖТ  `exception/` packages under the base package, with:
   188тЖТ   188тЖТ  - A `GlobalExceptionHandler` using `@RestControllerAdvice`
   189тЖТ   189тЖТ  - A sample `ApiResponse<T>` wrapper DTO
   190тЖТ   190тЖТ  - A sample health-check controller wired to the chosen dependencies (e.g. if `data-jpa` is
   191тЖТ   191тЖТ    selected, include a minimal `Entity`/`Repository`/`Service`/`Controller` CRUD vertical
   192тЖТ   192тЖТ    slice as a working example, not just empty folders)
   193тЖТ   193тЖТ- `hexagonal`: `domain/`, `application/`, `adapter/in/web/`, `adapter/out/persistence/` тАФ
   194тЖТ   194тЖТ  ports-and-adapters skeleton with a sample use case.
   195тЖТ   195тЖТ- `none`: skip тАФ leave Initializr's default single-package layout untouched.
   196тЖТ   196тЖТ
   197тЖТ   197тЖТ**Extras** (multi-select, independent of architecture choice):
   198тЖТ   198тЖТ- `Dockerfile` тАФ multi-stage build (build stage with the right Maven/Gradle image matching
   199тЖТ   199тЖТ  chosen Java version, slim JRE runtime stage), respects `packaging` (Jar/War).
   200тЖТ   200тЖТ- `docker-compose.yml` тАФ service block for the app + conditionally a `postgres`/`mysql`/
   201тЖТ   201тЖТ  `redis` service *if* the corresponding Initializr dependency was selected (detect from
   202тЖТ   202тЖТ  `dependencies` list: `data-jpa`+`postgresql` driver тЖТ add postgres service, etc.).
   203тЖТ   203тЖТ- GitHub Actions CI (`.github/workflows/ci.yml`) тАФ build + test workflow matching the chosen
   204тЖТ   204тЖТ  build tool (`mvn -B verify` or `./gradlew build`), with dependency caching configured
   205тЖТ   205тЖТ  correctly per build tool.
   206тЖТ   206тЖТ- `application-dev.yml` / `application-prod.yml` тАФ Spring profiles split out from the default
   207тЖТ   207тЖТ  `application.properties`, with profile activation wired via `SPRING_PROFILES_ACTIVE` and a
   208тЖТ   208тЖТ  sensible dev-vs-prod default (e.g. dev uses H2/local DB, prod reads from env vars).
   209тЖТ   209тЖТ- `.editorconfig`, a `.gitignore` tuned for the chosen build tool (if Initializr's default is
   210тЖТ   210тЖТ  insufficient), and a generated `README.md` with actual project name, run instructions
   211тЖТ   211тЖТ  matching the build tool, and Docker instructions if Docker extra was selected.
   212тЖТ   212тЖТ
   213тЖТ   213тЖТ**Template engine mechanics**:
   214тЖТ   214тЖТ- Use a real templating crate (`tera` or `minijinja` тАФ prefer `minijinja` for smaller
   215тЖТ   215тЖТ  footprint and no unsafe macro magic) rather than hand-rolled string replace, since files
   216тЖТ   216тЖТ  need conditionals (include a block only if a dependency is present) and loops (dependency
   217тЖТ   217тЖТ  list in README).
   218тЖТ   218тЖТ- All templates live in `springup-templates/assets/` and are embedded into the binary at
   219тЖТ   219тЖТ  compile time via `rust-embed`, so the shipped binary is fully self-contained тАФ no runtime
   220тЖТ   220тЖТ  fetch needed for the custom layer (only the Initializr base project requires network).
   221тЖТ   221тЖТ- Template context = a serializable view derived from `ProjectPlan` (package name, whether
   222тЖТ   222тЖТ  each relevant dependency is present, build tool, etc.) тАФ pass this into every render call.
   223тЖТ   223тЖТ
   224тЖТ   224тЖТ### 3.6 `springup.toml` тАФ the project manifest
   225тЖТ   225тЖТ
   226тЖТ   226тЖТWritten to the root of every generated project. This is the `package.json` equivalent тАФ
   227тЖТ   227тЖТit's what makes a future `springup add <module>` possible without re-deriving context.
   228тЖТ   228тЖТ
   229тЖТ   229тЖТ```toml
   230тЖТ   230тЖТ[project]
   231тЖТ   231тЖТspringup_version = "0.1.0"
   232тЖТ   232тЖТgenerated_at = "2026-07-02T10:00:00Z"
   233тЖТ   233тЖТgroup_id = "dev.raghavarora"
   234тЖТ   234тЖТartifact_id = "my-service"
   235тЖТ   235тЖТpackage_name = "dev.raghavarora.myservice"
   236тЖТ   236тЖТspring_boot_version = "3.5.0"
   237тЖТ   237тЖТbuild_tool = "maven"
   238тЖТ   238тЖТlanguage = "java"
   239тЖТ   239тЖТjava_version = "21"
   240тЖТ   240тЖТ
   241тЖТ   241тЖТ[architecture]
   242тЖТ   242тЖТkind = "layered"
   243тЖТ   243тЖТ
   244тЖТ   244тЖТ[extras]
   245тЖТ   245тЖТenabled = ["docker", "docker-compose", "github-actions-ci", "config-profiles"]
   246тЖТ   246тЖТ
   247тЖТ   247тЖТ[dependencies]
   248тЖТ   248тЖТinitializr = ["web", "data-jpa", "postgresql", "validation", "security"]
   249тЖТ   249тЖТ```
   250тЖТ   250тЖТ
   251тЖТ   251тЖТThis file is inert metadata for v1 (the tool doesn't read it back for anything beyond a
   252тЖТ   252тЖТfuture `add` command), but it must be written correctly and documented, since it's the
   253тЖТ   253тЖТextension point for everything that comes after v1.

   257тЖТ   257тЖТ## 4. CLI surface (commands & flags)
   258тЖТ   258тЖТ
   259тЖТ   259тЖТUse `clap` v4 with derive macros. Design the command tree now even though only `new` is
   260тЖТ   260тЖТrequired for v1, so help text and completions are coherent from day one.
   261тЖТ   261тЖТ
   262тЖТ   262тЖТ```
   263тЖТ   263тЖТspringup new [NAME]                 Scaffold a new project (interactive if flags omitted)
   264тЖТ   264тЖТspringup add <MODULE>                [v2+, stub with "not yet implemented" for v1]
   265тЖТ   265тЖТspringup config get|set|list         Manage global user config
   266тЖТ   266тЖТspringup completions <SHELL>         Generate shell completion script (bash/zsh/fish/powershell)
   267тЖТ   267тЖТspringup update                      Self-update the binary to latest release
   268тЖТ   268тЖТspringup --version / -V
   269тЖТ   269тЖТspringup --help / -h
   270тЖТ   270тЖТ```
   271тЖТ   271тЖТ
   272тЖТ   272тЖТ### `springup new` flags (non-interactive mode)
   273тЖТ   273тЖТ
   274тЖТ   274тЖТ```
   275тЖТ   275тЖТ-n, --name <NAME>
   276тЖТ   276тЖТ    --group-id <GROUP_ID>              default from global config or "com.example"
   277тЖТ   277тЖТ    --artifact-id <ARTIFACT_ID>
   278тЖТ   278тЖТ    --description <DESC>
   279тЖТ   279тЖТ    --boot-version <VERSION>           default: latest stable per Initializr metadata
   280тЖТ   280тЖТ    --build-tool <maven|gradle|gradle-kotlin>
   281тЖТ   281тЖТ    --language <java|kotlin>
   282тЖТ   282тЖТ    --java-version <VERSION>
   283тЖТ   283тЖТ    --packaging <jar|war>
   284тЖТ   284тЖТ-d, --deps <DEP,DEP,...>               comma-separated Initializr dependency ids
   285тЖТ   285тЖТ    --architecture <none|layered|hexagonal>
   286тЖТ   286тЖТ    --extras <EXTRA,EXTRA,...>         docker,docker-compose,ci,config-profiles
   287тЖТ   287тЖТ-o, --output <DIR>                     default: ./<artifact-id>
   288тЖТ   288тЖТ    --git / --no-git                   default: --git
   289тЖТ   289тЖТ    --commit / --no-commit             default: --no-commit
   290тЖТ   290тЖТ-y, --yes                              accept all defaults, skip wizard entirely
   291тЖТ   291тЖТ    --dry-run                          print the resolved ProjectPlan as JSON/TOML, generate nothing
   292тЖТ   292тЖТ```
   293тЖТ   293тЖТ
   294тЖТ   294тЖТ**Interaction mode resolution rule** (this must be unambiguous, write it as actual logic, not
   295тЖТ   295тЖТvibes): if stdin is not a TTY, OR any of `--yes`/`-y` is passed, OR enough required flags are
   296тЖТ   296тЖТpresent to fully resolve a `ProjectPlan` with no ambiguity, skip the wizard and go straight to
   297тЖТ   297тЖТgeneration, filling any *non-required* gaps with global-config defaults or hardcoded sane
   298тЖТ   298тЖТdefaults. If flags are partially given but insufficient, run the wizard but pre-fill/skip the
   299тЖТ   299тЖТsteps already answered by flags (this is the "best of both" UX тАФ don't force an all-or-nothing
   300тЖТ   300тЖТchoice between flags and wizard).
   301тЖТ   301тЖТ
   302тЖТ   302тЖТ---
   303тЖТ   303тЖТ
   304тЖТ   304тЖТ## 5. Interactive wizard UX requirements
   305тЖТ   305тЖТ
   306тЖТ   306тЖТ- Library choice: `dialoguer` (simpler, more standard for this style of prompt-by-prompt
   307тЖТ   307тЖТ  wizard than a full `ratatui` app) is likely the right call тАФ this is a sequence of
   308тЖТ   308тЖТ  select/multi-select/input prompts, not a persistent dashboard. Use `ratatui` only if you
   309тЖТ   309тЖТ  want a genuinely different, more visual experience (e.g. a dependency-browser split pane);
   310тЖТ   310тЖТ  otherwise `dialoguer` + `indicatif` (spinners/progress bars) is simpler and battle-tested.
   311тЖТ   311тЖТ- Every prompt needs: clear question text, sensible pre-selected default, `?`-triggerable
   312тЖТ   312тЖТ  inline help/description where non-obvious (e.g. showing each dependency's one-line
   313тЖТ   313тЖТ  description from Initializr metadata next to it in the multi-select).
   314тЖТ   314тЖТ- Dependency multi-select must be **searchable/filterable** (type to filter the list) тАФ with
   315тЖТ   315тЖТ  60+ Initializr dependencies, an unfiltered scroll list is a bad experience. Group by
   316тЖТ   316тЖТ  Initializr's own categories (Web, SQL, NoSQL, Security, Messaging, Ops, etc.) with category
   317тЖТ   317тЖТ  headers in the list.
   318тЖТ   318тЖТ- Show a spinner during network calls (metadata fetch, project generation download) with
   319тЖТ   319тЖТ  elapsed time, and a clear failure state with retry guidance.
   320тЖТ   320тЖТ- End with a colorized summary: what was generated, file tree (top 2 levels), and copy-paste
   321тЖТ   321тЖТ  next-step commands (`cd`, build command matching build tool, run command).
   322тЖТ   322тЖТ- Respect `NO_COLOR` env var and `--no-color` flag. Respect `CLICOLOR_FORCE`.
   323тЖТ   323тЖТ- All user-facing strings centralized (not scattered `println!`) so the tool could add i18n
   324тЖТ   324тЖТ  later without a rewrite тАФ a simple `messages.rs` module is enough for v1, no need for a
   325тЖТ   325тЖТ  full i18n crate.
   326тЖТ   326тЖТ

   329тЖТ   329тЖТ## 6. Production-grade engineering requirements
   330тЖТ   330тЖТ
   331тЖТ   331тЖТThese are non-negotiable for calling this "production grade" тАФ treat as a checklist.
   332тЖТ   332тЖТ
   333тЖТ   333тЖТ### 6.1 Error handling
   334тЖТ   334тЖТ- `thiserror` for typed library errors in `springup-core` (each failure mode is a distinct
   335тЖТ   335тЖТ  variant: `InitializrUnreachable`, `InvalidDependency { id, suggestion }`,
   336тЖТ   336тЖТ  `TemplateRenderError`, `IoError`, `InvalidProjectMetadata`, etc.).
   337тЖТ   337тЖТ- `anyhow` (or `color-eyre` for prettier terminal output) at the `springup-cli` boundary for
   338тЖТ   338тЖТ  ergonomic error propagation and nicely formatted terminal error reports with backtraces in
   339тЖТ   339тЖТ  `--verbose` mode.
   340тЖТ   340тЖТ- Never `unwrap()`/`panic!()` on any path reachable from user input or network I/O. `unwrap()`
   341тЖТ   341тЖТ  is acceptable only on genuinely-infallible internal invariants, and even then prefer
   342тЖТ   342тЖТ  `expect("reason")` documenting why it's infallible.
   343тЖТ   343тЖТ- Every network call has a timeout (`reqwest` client configured with sane connect/read
   344тЖТ   344тЖТ  timeouts) and a bounded retry with backoff for transient failures (`backoff` crate or
   345тЖТ   345тЖТ  hand-rolled exponential backoff, 3 attempts max).
   346тЖТ   346тЖТ
   347тЖТ   347тЖТ### 6.2 Logging & diagnostics
   348тЖТ   348тЖТ- `tracing` + `tracing-subscriber` for structured logging. `--verbose`/`-v` (repeatable, `-vv`
   349тЖТ   349тЖТ  for debug, `-vvv` for trace) controls log level. Default: only user-facing progress output,
   350тЖТ   350тЖТ  no log noise.
   351тЖТ   351тЖТ- `--dry-run` and `--verbose` should make it possible to fully understand what the tool *would
   352тЖТ   352тЖТ  do* without needing to read source.
   353тЖТ   353тЖТ
   354тЖТ   354тЖТ### 6.3 Config management (global user config)
   355тЖТ   355тЖТ- Location via the `directories` crate (`ProjectDirs::from("dev", "springup", "springup")`)
   356тЖТ   356тЖТ  тЖТ e.g. `~/.config/springup/config.toml` on Linux, correct platform equivalents on
   357тЖТ   357тЖТ  macOS/Windows.
   358тЖТ   358тЖТ- Stores: default group-id, default author/org for READMEs, default Java version, default
   359тЖТ   359тЖТ  build tool, preferred package manager mirror (if ever needed), telemetry opt-in flag
   360тЖТ   360тЖТ  (default OFF тАФ see ┬з6.7), color preference.
   361тЖТ   361тЖТ- `springup config set group-id dev.raghavarora`, `springup config get group-id`,
   362тЖТ   362тЖТ  `springup config list` тАФ implement properly, not just a TODO.
   363тЖТ   363тЖТ- Precedence, most specific wins: CLI flags > project-local `.springuprc.toml` (if present in
   364тЖТ   364тЖТ  cwd) > global user config > hardcoded defaults. Document this precedence in `--help` and
   365тЖТ   365тЖТ  README.
   366тЖТ   366тЖТ
   367тЖТ   367тЖТ### 6.4 Testing
   368тЖТ   368тЖТ- Unit tests in `springup-core` for: `ProjectPlan` validation logic, template context
   369тЖТ   369тЖТ  derivation, dependency-id fuzzy matching, manifest serialization round-trip.
   370тЖТ   370тЖТ- Integration tests in `tests/` that actually run the compiled binary (`assert_cmd` crate)
   371тЖТ   371тЖТ  against a **mocked** Initializr server (`wiremock` or `mockito` crate) тАФ never hit the real
   372тЖТ   372тЖТ  network in CI. Assert on generated file tree structure and key file contents (e.g. Dockerfile
   373тЖТ   373тЖТ  contains the right base image tag for the chosen Java version).
   374тЖТ   374тЖТ- Golden-file / snapshot tests (`insta` crate) for generated file contents (Dockerfile,
   375тЖТ   375тЖТ  GitHub Actions YAML, README) so template regressions are caught immediately on any template
   376тЖТ   376тЖТ  edit.
   377тЖТ   377тЖТ- A real (non-mocked) opt-in "smoke test" behind a feature flag or separate CI job that hits
   378тЖТ   378тЖТ  the live Initializr API once, to catch upstream API drift тАФ run on a schedule (e.g. nightly
   379тЖТ   379тЖТ  GitHub Actions cron), not on every PR.
   380тЖТ   380тЖТ
   381тЖТ   381тЖТ### 6.5 CI/CD for the tool itself
   382тЖТ   382тЖТ- GitHub Actions workflow(s):
   383тЖТ   383тЖТ  - `ci.yml`: on every PR/push тАФ `cargo fmt --check`, `cargo clippy -- -D warnings`,
   384тЖТ   384тЖТ    `cargo test --all-features`, run on a matrix of `ubuntu-latest`, `macos-latest`,
   385тЖТ   385тЖТ    `windows-latest`.
   386тЖТ   386тЖТ  - `release.yml`: on tag push (`v*`) тАФ build release binaries for at minimum:
   387тЖТ   387тЖТ    `x86_64-unknown-linux-gnu`, `x86_64-unknown-linux-musl` (static, for broad Linux compat),
   388тЖТ   388тЖТ    `aarch64-unknown-linux-gnu`, `x86_64-apple-darwin`, `aarch64-apple-darwin`,
   389тЖТ   389тЖТ    `x86_64-pc-windows-msvc`. Use `cargo-dist` (recommended тАФ it generates the release
   390тЖТ   390тЖТ    workflow, install scripts, and Homebrew/Scoop manifests for you) rather than hand-rolling
   391тЖТ   391тЖТ    cross-compilation matrix YAML from scratch.
   392тЖТ   392тЖТ- `cargo-dist` also generates the `curl | sh` install script and can publish a Homebrew tap
   393тЖТ   393тЖТ  and Scoop manifest automatically тАФ use it rather than reinventing this.
   394тЖТ   394тЖТ- Also publish to crates.io (`cargo publish`) as a secondary install path for Rust users who
   395тЖТ   395тЖТ  prefer `cargo install springup`.
   396тЖТ   396тЖТ
   397тЖТ   397тЖТ### 6.6 Documentation
   398тЖТ   398тЖТ- Top-level `README.md`: install instructions for every distribution channel, quickstart GIF
   399тЖТ   399тЖТ  or asciinema-style terminal recording reference, full command reference, architecture
   400тЖТ   400тЖТ  overview linking to `ARCHITECTURE.md`.
   401тЖТ   401тЖТ- `ARCHITECTURE.md`: the module layout from ┬з3.2, the `ProjectPlan` data flow, how to add a
   402тЖТ   402тЖТ  new "extra" template.
   403тЖТ   403тЖТ- `CONTRIBUTING.md`: how to run tests, how to add a new architecture skeleton or extra, PR
   404тЖТ   404тЖТ  checklist (fmt/clippy/tests must pass).
   405тЖТ   405тЖТ- Every public function/struct in `springup-core` has doc comments (`cargo doc` should
   406тЖТ   406тЖТ  produce something genuinely useful, not empty stubs).
   407тЖТ   407тЖТ
   408тЖТ   408тЖТ### 6.7 Privacy / telemetry
   409тЖТ   409тЖТ- If any telemetry is ever added (usage analytics), it must be **opt-in**, off by default,
   410тЖТ   410тЖТ  clearly disclosed in README, and disableable via `springup config set telemetry false` and
   411тЖТ   411тЖТ  an env var (`SPRINGUP_TELEMETRY=0`). For v1, ship with **no telemetry at all** тАФ simplest
   412тЖТ   412тЖТ  and most trustworthy default; mention this explicitly in README as a stated principle.
   413тЖТ   413тЖТ
   414тЖТ   414тЖТ### 6.8 Security
   415тЖТ   415тЖТ- Never execute arbitrary code from downloaded templates (the Initializr zip is data, not
   416тЖТ   416тЖТ  code тАФ treat it as such; don't `eval`/shell-exec anything extracted from it).
   417тЖТ   417тЖТ- Validate zip extraction against path traversal (zip-slip vulnerability) тАФ use a
   418тЖТ   418тЖТ  well-maintained zip crate (`zip` crate) and explicitly sanitize/verify every extracted
   419тЖТ   419тЖТ  path stays within the target directory before writing.
   420тЖТ   420тЖТ- Pin dependency versions in `Cargo.lock`, commit it (this is a binary, not a library others
   421тЖТ   421тЖТ  depend on тАФ lockfile should always be committed).
   422тЖТ   422тЖТ- Run `cargo audit` in CI as a scheduled job to catch known vulnerabilities in dependencies.
   423тЖТ   423тЖТ
   424тЖТ   424тЖТ### 6.9 Performance
   425тЖТ   425тЖТ- Cold start (no cache) to first wizard prompt should feel instant (<100ms) тАФ metadata fetch
   426тЖТ   426тЖТ  happens *after* the prompts that don't need it (project name, output dir) start, not
   427тЖТ   427тЖТ  before, so perceived latency is hidden. Fetch metadata concurrently with the first 1-2
   428тЖТ   428тЖТ  wizard prompts using an async task, `await` it only when the wizard reaches the
   429тЖТ   429тЖТ  dependency-selection step.
   430тЖТ   430тЖТ- Use `tokio` (async runtime, since we're doing concurrent network + prompt work) with a
   431тЖТ   431тЖТ  minimal feature set (`rt-multi-thread`, `macros`, `time` тАФ not the full `full` feature
   432тЖТ   432тЖТ  flag, to keep binary size and compile time down).
   433тЖТ   433тЖТ- Release profile: enable LTO and strip symbols in `Cargo.toml` `[profile.release]`
   434тЖТ   434тЖТ  (`lto = true`, `strip = true`, `codegen-units = 1`) for smaller, faster binaries тАФ the
   435тЖТ   435тЖТ  build-time cost is worth it for a tool distributed as prebuilt binaries.
   436тЖТ   436тЖТ
