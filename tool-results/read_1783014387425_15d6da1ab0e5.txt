     1‚Üí# Build Spec: `springup` ‚Äî A Production-Grade Spring Boot Project Scaffolder in Rust
     2‚Üí
     3‚Üí> Hand this entire document to an AI coding assistant (Claude Code, Cursor, etc.) as the
     4‚Üí> primary instruction set. It describes a complete, installable CLI tool ‚Äî not a toy script.
     5‚Üí> Sections are ordered so an agent can work top-to-bottom and produce a working `v0.1.0`,
     6‚Üí> then iterate toward the full feature set.
     7‚Üí
     8‚Üí---
     9‚Üí
    10‚Üí## 0. One-line pitch
    11‚Üí
    12‚Üí`springup` is what `npm create`, `bun create`, and `cargo generate` are to their ecosystems,
    13‚Üíbut for Spring Boot: a fast, single static binary that interactively (or non-interactively)
    14‚Üíscaffolds a production-ready Spring Boot backend by combining the official Spring Initializr
    15‚ÜíAPI with an opinionated custom template layer (architecture skeletons, Docker, CI, config
    16‚Üíprofiles) ‚Äî written in Rust, distributed as a real CLI tool.
    17‚Üí
    18‚Üí---
    19‚Üí
    20‚Üí## 1. Goals & non-goals
    21‚Üí
    22‚Üí**Goals**
    23‚Üí- Feel as polished and fast as `bun create` / `create-t3-app`: sub-second startup, smooth
    24‚Üí  interactive prompts, sensible defaults, zero Java/Maven/Gradle required to just *generate*
    25‚Üí  a project.
    26‚Üí- Two first-class modes: interactive TUI wizard (default) and fully flag-driven non-interactive
    27‚Üí  mode (for CI, scripting, muscle-memory power users).
    28‚Üí- Generate real, buildable Spring Boot projects by delegating dependency/version resolution to
    29‚Üí  the official Spring Initializr API (`https://start.spring.io`), then layering custom
    30‚Üí  extras on top (architecture skeleton, Dockerfile, GitHub Actions, application.yml profiles).
    31‚Üí- Be genuinely production-grade as a piece of software: proper error handling, logging,
    32‚Üí  tests, CI, versioned releases, cross-platform binaries, shell completions, self-update.
    33‚Üí- Support an "add" workflow later (`springup add module`) ‚Äî so today's design must not paint
    34‚Üí  the tool into a one-shot-generator corner. Use a lightweight project config file
    35‚Üí  (`springup.toml`) written into generated projects to enable this later without a rewrite.
    36‚Üí
    37‚Üí**Non-goals (explicitly out of scope for v1)**
    38‚Üí- No frontend/monorepo scaffolding (no Next.js/React generation). Backend + Docker/CI only.
    39‚Üí- No bundling of a JVM, Maven, or Gradle. The tool never requires Java to be installed to
    40‚Üí  *generate* a project (it may shell out to `./mvnw`/`./gradlew` only for optional post-gen
    41‚Üí  steps like "run tests now?", which are opt-in and gracefully skipped if unavailable).
    42‚Üí- No GUI. Terminal only.
    43‚Üí- No plugin marketplace/registry in v1 (design for it, don't build it yet).
    44‚Üí
    45‚Üí---
    46‚Üí
    47‚Üí## 2. Prior art to study before writing code
    48‚Üí
    49‚ÜíThe agent should mentally (or actually, if it has web access) benchmark against:
    50‚Üí- `cargo-generate` (Rust) ‚Äî template mechanics, `{{placeholder}}` substitution patterns.
    51‚Üí- `create-t3-app` (TS) ‚Äî interactive prompt UX, layered "optional add-on" selection.
    52‚Üí- `starship` (Rust) ‚Äî how a beloved Rust CLI handles config file precedence, cross-platform
    53‚Üí  installs, and shell completions.
    54‚Üí- `bun create` / `npm create` ‚Äî speed expectations and flag ergonomics.
    55‚Üí- Spring Initializr itself (`start.spring.io`) ‚Äî this tool is a client of its REST API, not a
    56‚Üí  replacement for it. Read its API docs (`https://start.spring.io` returns machine-readable
    57‚Üí  metadata at the root when queried with `Accept: application/json`).
    58‚Üí

    60‚Üí
    61‚Üí## 3. Core architecture
    62‚Üí
    63‚Üí### 3.1 High-level flow
    64‚Üí
    65‚Üí```
    66‚ÜíUser runs `springup new`
    67‚Üí        ‚îÇ
    68‚Üí        ‚ñº
    69‚Üí[Mode detection] ‚îÄ‚îÄ flags present & non-interactive? ‚îÄ‚îÄ‚ñ∫ skip wizard, validate flags
    70‚Üí        ‚îÇ no
    71‚Üí        ‚ñº
    72‚Üí[Interactive Wizard] (ratatui or dialoguer-based prompts)
    73‚Üí   1. Project metadata (group, artifact, name, package, description)
    74‚Üí   2. Spring Boot version (fetched live from Initializr metadata, cached)
    75‚Üí   3. Build tool: Maven | Gradle (Groovy) | Gradle (Kotlin)
    76‚Üí   4. Language: Java | Kotlin
    77‚Üí   5. Java version
    78‚Üí   6. Packaging: Jar | War
    79‚Üí   7. Dependencies (multi-select, searchable, grouped by category ‚Äî from Initializr metadata)
    80‚Üí   8. [Custom layer] Architecture skeleton? (none / layered / hexagonal) ‚Äî optional, user choice
    81‚Üí   9. [Custom layer] Extras multi-select: Dockerfile, docker-compose.yml, GitHub Actions CI,
    82‚Üí      application-{dev,prod}.yml profiles, .editorconfig, pre-commit hook, README badges
    83‚Üí        ‚îÇ
    84‚Üí        ‚ñº
    85‚Üí[Resolve & Plan] ‚Äî build a ProjectPlan struct (see ¬ß3.3), validate combos
    86‚Üí        ‚îÇ
    87‚Üí        ‚ñº
    88‚Üí[Fetch base project from Initializr] (zip stream) ‚îÄ‚îÄ‚ñ∫ extract to temp dir
    89‚Üí        ‚îÇ
    90‚Üí        ‚ñº
    91‚Üí[Apply custom template layer] ‚Äî inject architecture folders, Docker/CI files,
    92‚Üí   rewrite application.yml, write springup.toml manifest
    93‚Üí        ‚îÇ
    94‚Üí        ‚ñº
    95‚Üí[Post-gen] ‚Äî git init (optional), initial commit (optional), print summary + next steps
    96‚Üí```
    97‚Üí
    98‚Üí### 3.2 Crate/module layout
    99‚Üí
   100‚Üí```
   101‚Üíspringup/
   102‚Üí‚îú‚îÄ‚îÄ Cargo.toml
   103‚Üí‚îú‚îÄ‚îÄ crates/
   104‚Üí‚îÇ   ‚îú‚îÄ‚îÄ springup-cli/          # binary crate ‚Äî arg parsing, wizard orchestration, output
   105‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/main.rs
   106‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/cli.rs         # clap definitions
   107‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/wizard/        # interactive prompt flows
   108‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/commands/      # new.rs, add.rs, config.rs, completions.rs, update.rs
   109‚Üí‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ src/ui/            # terminal rendering helpers, spinners, theming
   110‚Üí‚îÇ   ‚îú‚îÄ‚îÄ springup-core/         # library crate ‚Äî pure logic, fully unit-testable, no I/O-heavy CLI concerns
   111‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/plan.rs        # ProjectPlan struct + validation
   112‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/initializr/    # Spring Initializr API client + response models
   113‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/template/      # custom template engine (rendering, placeholder substitution)
   114‚Üí‚îÇ   ‚îÇ   ‚îú‚îÄ‚îÄ src/manifest.rs    # springup.toml read/write (project-level state)
   115‚Üí‚îÇ   ‚îÇ   ‚îî‚îÄ‚îÄ src/config.rs      # global user config (~/.config/springup/config.toml)
   116‚Üí‚îÇ   ‚îî‚îÄ‚îÄ springup-templates/    # embedded custom templates (architecture skeletons, CI, Docker)
   117‚Üí‚îÇ       ‚îî‚îÄ‚îÄ assets/
   118‚Üí‚îÇ           ‚îú‚îÄ‚îÄ architectures/layered/
   119‚Üí‚îÇ           ‚îú‚îÄ‚îÄ architectures/hexagonal/
   120‚Üí‚îÇ           ‚îú‚îÄ‚îÄ docker/
   121‚Üí‚îÇ           ‚îú‚îÄ‚îÄ ci/github-actions/
   122‚Üí‚îÇ           ‚îî‚îÄ‚îÄ config-profiles/
   123‚Üí‚îú‚îÄ‚îÄ xtask/                     # cargo-xtask pattern: release automation, template asset checks
   124‚Üí‚îî‚îÄ‚îÄ tests/                     # integration tests (see ¬ß8)
   125‚Üí```
   126‚Üí
   127‚ÜíRationale for splitting `springup-core` out: it must be usable without a terminal (unit
   128‚Üítests, potential future library consumers, WASM playground, etc.) and keeps `main.rs` thin.
   129‚Üí
   130‚Üí### 3.3 Core data model
   131‚Üí
   132‚Üí```rust
   133‚Üí/// The fully-resolved, validated description of what to generate.
   134‚Üí/// Built by EITHER the wizard OR flag parsing ‚Äî both paths converge here.
   135‚Üípub struct ProjectPlan {
   136‚Üí    pub group_id: String,
   137‚Üí    pub artifact_id: String,
   138‚Üí    pub name: String,
   139‚Üí    pub description: String,
   140‚Üí    pub package_name: String,
   141‚Üí    pub spring_boot_version: String,
   142‚Üí    pub build_tool: BuildTool,       // Maven | GradleGroovy | GradleKotlin
   143‚Üí    pub language: Language,          // Java | Kotlin
   144‚Üí    pub java_version: String,
   145‚Üí    pub packaging: Packaging,        // Jar | War
   146‚Üí    pub dependencies: Vec<DependencyId>,   // Initializr dependency ids, e.g. "web", "data-jpa"
   147‚Üí    pub architecture: Option<ArchitectureKind>, // None | Layered | Hexagonal
   148‚Üí    pub extras: Vec<ExtraFeature>,   // Docker, DockerCompose, GithubActionsCi, ConfigProfiles, ...
   149‚Üí    pub output_dir: PathBuf,
   150‚Üí    pub git_init: bool,
   151‚Üí    pub initial_commit: bool,
   152‚Üí}
   153‚Üí```
   154‚Üí
   155‚ÜíKeep this struct serializable (`serde`). It doubles as the schema for:
   156‚Üí- Non-interactive flags (`clap` derives directly into a builder for this struct).
   157‚Üí- A future `--from-file plan.toml` mode (batch/reproducible generation ‚Äî trivial to add
   158‚Üí  later since the plan is already data, not scattered logic).
   159‚Üí
   160‚Üí### 3.4 Spring Initializr integration
   161‚Üí
   162‚Üí- Base URL: `https://start.spring.io`.
   163‚Üí- On startup (or on a `springup update-metadata` command / cached with TTL), fetch
   164‚Üí  `GET https://start.spring.io` with `Accept: application/json` to get the full metadata
   165‚Üí  document: available Spring Boot versions, dependency list with categories/descriptions/
   166‚Üí  compatibility ranges, Java versions, packaging options, language options.
   167‚Üí- Cache this metadata locally (`~/.cache/springup/initializr-metadata.json` or platform
   168‚Üí  equivalent via the `directories` crate) with a TTL (e.g. 24h) so the wizard is instant on
   169‚Üí  repeat runs and works offline using stale-but-usable cached data (warn the user it's stale).
   170‚Üí- To generate the actual project, call
   171‚Üí  `GET https://start.spring.io/starter.zip?type=...&language=...&bootVersion=...&groupId=...&artifactId=...&name=...&packageName=...&packaging=...&javaVersion=...&dependencies=...`
   172‚Üí  and stream the zip response to a temp file, then extract.
   173‚Üí- Handle Initializr downtime/errors gracefully: clear error message, suggest retry, never
   174‚Üí  panic. Consider a `--offline` flag that only works if a previously-downloaded base template
   175‚Üí  cache exists (stretch goal, not required for v1).
   176‚Üí- Validate dependency IDs against fetched metadata *before* hitting the generate endpoint, so
   177‚Üí  users get a fast, friendly "unknown dependency 'foo', did you mean 'flyway'?" instead of a
   178‚Üí  raw API error. Implement fuzzy matching (`strsim` crate, e.g. Jaro-Winkler) for suggestions.
   179‚Üí
   180‚Üí### 3.5 Custom template layer (the "extras" system)
   181‚Üí
   182‚ÜíThis is what differentiates `springup` from a bare Initializr CLI wrapper. Runs *after* the
   183‚Üíbase project is extracted.
   184‚Üí
   185‚Üí**Architecture skeletons** (optional, chosen interactively per ¬ß3.1 step 8):
   186‚Üí- `layered`: generates `controller/`, `service/`, `repository/`, `dto/`, `entity/`,
   187‚Üí  `exception/` packages under the base package, with:
   188‚Üí  - A `GlobalExceptionHandler` using `@RestControllerAdvice`
   189‚Üí  - A sample `ApiResponse<T>` wrapper DTO
   190‚Üí  - A sample health-check controller wired to the chosen dependencies (e.g. if `data-jpa` is
   191‚Üí    selected, include a minimal `Entity`/`Repository`/`Service`/`Controller` CRUD vertical
   192‚Üí    slice as a working example, not just empty folders)
   193‚Üí- `hexagonal`: `domain/`, `application/`, `adapter/in/web/`, `adapter/out/persistence/` ‚Äî
   194‚Üí  ports-and-adapters skeleton with a sample use case.
   195‚Üí- `none`: skip ‚Äî leave Initializr's default single-package layout untouched.
   196‚Üí
   197‚Üí**Extras** (multi-select, independent of architecture choice):
   198‚Üí- `Dockerfile` ‚Äî multi-stage build (build stage with the right Maven/Gradle image matching
   199‚Üí  chosen Java version, slim JRE runtime stage), respects `packaging` (Jar/War).
   200‚Üí- `docker-compose.yml` ‚Äî service block for the app + conditionally a `postgres`/`mysql`/
   201‚Üí  `redis` service *if* the corresponding Initializr dependency was selected (detect from
   202‚Üí  `dependencies` list: `data-jpa`+`postgresql` driver ‚Üí add postgres service, etc.).
   203‚Üí- GitHub Actions CI (`.github/workflows/ci.yml`) ‚Äî build + test workflow matching the chosen
   204‚Üí  build tool (`mvn -B verify` or `./gradlew build`), with dependency caching configured
   205‚Üí  correctly per build tool.
   206‚Üí- `application-dev.yml` / `application-prod.yml` ‚Äî Spring profiles split out from the default
   207‚Üí  `application.properties`, with profile activation wired via `SPRING_PROFILES_ACTIVE` and a
   208‚Üí  sensible dev-vs-prod default (e.g. dev uses H2/local DB, prod reads from env vars).
   209‚Üí- `.editorconfig`, a `.gitignore` tuned for the chosen build tool (if Initializr's default is
   210‚Üí  insufficient), and a generated `README.md` with actual project name, run instructions
   211‚Üí  matching the build tool, and Docker instructions if Docker extra was selected.
   212‚Üí
   213‚Üí**Template engine mechanics**:
   214‚Üí- Use a real templating crate (`tera` or `minijinja` ‚Äî prefer `minijinja` for smaller
   215‚Üí  footprint and no unsafe macro magic) rather than hand-rolled string replace, since files
   216‚Üí  need conditionals (include a block only if a dependency is present) and loops (dependency
   217‚Üí  list in README).
   218‚Üí- All templates live in `springup-templates/assets/` and are embedded into the binary at
   219‚Üí  compile time via `rust-embed`, so the shipped binary is fully self-contained ‚Äî no runtime
   220‚Üí  fetch needed for the custom layer (only the Initializr base project requires network).
   221‚Üí- Template context = a serializable view derived from `ProjectPlan` (package name, whether
   222‚Üí  each relevant dependency is present, build tool, etc.) ‚Äî pass this into every render call.
   223‚Üí
   224‚Üí### 3.6 `springup.toml` ‚Äî the project manifest
   225‚Üí
   226‚ÜíWritten to the root of every generated project. This is the `package.json` equivalent ‚Äî
   227‚Üíit's what makes a future `springup add <module>` possible without re-deriving context.
   228‚Üí
   229‚Üí```toml
   230‚Üí[project]
   231‚Üíspringup_version = "0.1.0"
   232‚Üígenerated_at = "2026-07-02T10:00:00Z"
   233‚Üígroup_id = "dev.raghavarora"
   234‚Üíartifact_id = "my-service"
   235‚Üípackage_name = "dev.raghavarora.myservice"
   236‚Üíspring_boot_version = "3.5.0"
   237‚Üíbuild_tool = "maven"
   238‚Üílanguage = "java"
   239‚Üíjava_version = "21"
   240‚Üí
   241‚Üí[architecture]
   242‚Üíkind = "layered"
   243‚Üí
   244‚Üí[extras]
   245‚Üíenabled = ["docker", "docker-compose", "github-actions-ci", "config-profiles"]
   246‚Üí
   247‚Üí[dependencies]
   248‚Üíinitializr = ["web", "data-jpa", "postgresql", "validation", "security"]
   249‚Üí```
   250‚Üí
   251‚ÜíThis file is inert metadata for v1 (the tool doesn't read it back for anything beyond a
   252‚Üífuture `add` command), but it must be written correctly and documented, since it's the
   253‚Üíextension point for everything that comes after v1.
   254‚Üí

   257‚Üí## 4. CLI surface (commands & flags)
   258‚Üí
   259‚ÜíUse `clap` v4 with derive macros. Design the command tree now even though only `new` is
   260‚Üírequired for v1, so help text and completions are coherent from day one.
   261‚Üí
   262‚Üí```
   263‚Üíspringup new [NAME]                 Scaffold a new project (interactive if flags omitted)
   264‚Üíspringup add <MODULE>                [v2+, stub with "not yet implemented" for v1]
   265‚Üíspringup config get|set|list         Manage global user config
   266‚Üíspringup completions <SHELL>         Generate shell completion script (bash/zsh/fish/powershell)
   267‚Üíspringup update                      Self-update the binary to latest release
   268‚Üíspringup --version / -V
   269‚Üíspringup --help / -h
   270‚Üí```
   271‚Üí
   272‚Üí### `springup new` flags (non-interactive mode)
   273‚Üí
   274‚Üí```
   275‚Üí-n, --name <NAME>
   276‚Üí    --group-id <GROUP_ID>              default from global config or "com.example"
   277‚Üí    --artifact-id <ARTIFACT_ID>
   278‚Üí    --description <DESC>
   279‚Üí    --boot-version <VERSION>           default: latest stable per Initializr metadata
   280‚Üí    --build-tool <maven|gradle|gradle-kotlin>
   281‚Üí    --language <java|kotlin>
   282‚Üí    --java-version <VERSION>
   283‚Üí    --packaging <jar|war>
   284‚Üí-d, --deps <DEP,DEP,...>               comma-separated Initializr dependency ids
   285‚Üí    --architecture <none|layered|hexagonal>
   286‚Üí    --extras <EXTRA,EXTRA,...>         docker,docker-compose,ci,config-profiles
   287‚Üí-o, --output <DIR>                     default: ./<artifact-id>
   288‚Üí    --git / --no-git                   default: --git
   289‚Üí    --commit / --no-commit             default: --no-commit
   290‚Üí-y, --yes                              accept all defaults, skip wizard entirely
   291‚Üí    --dry-run                          print the resolved ProjectPlan as JSON/TOML, generate nothing
   292‚Üí```
   293‚Üí
   294‚Üí**Interaction mode resolution rule** (this must be unambiguous, write it as actual logic, not
   295‚Üívibes): if stdin is not a TTY, OR any of `--yes`/`-y` is passed, OR enough required flags are
   296‚Üípresent to fully resolve a `ProjectPlan` with no ambiguity, skip the wizard and go straight to
   297‚Üígeneration, filling any *non-required* gaps with global-config defaults or hardcoded sane
   298‚Üídefaults. If flags are partially given but insufficient, run the wizard but pre-fill/skip the
   299‚Üísteps already answered by flags (this is the "best of both" UX ‚Äî don't force an all-or-nothing
   300‚Üíchoice between flags and wizard).
   301‚Üí

   304‚Üí## 5. Interactive wizard UX requirements
   305‚Üí
   306‚Üí- Library choice: `dialoguer` (simpler, more standard for this style of prompt-by-prompt
   307‚Üí  wizard than a full `ratatui` app) is likely the right call ‚Äî this is a sequence of
   308‚Üí  select/multi-select/input prompts, not a persistent dashboard. Use `ratatui` only if you
   309‚Üí  want a genuinely different, more visual experience (e.g. a dependency-browser split pane);
   310‚Üí  otherwise `dialoguer` + `indicatif` (spinners/progress bars) is simpler and battle-tested.
   311‚Üí- Every prompt needs: clear question text, sensible pre-selected default, `?`-triggerable
   312‚Üí  inline help/description where non-obvious (e.g. showing each dependency's one-line
   313‚Üí  description from Initializr metadata next to it in the multi-select).
   314‚Üí- Dependency multi-select must be **searchable/filterable** (type to filter the list) ‚Äî with
   315‚Üí  60+ Initializr dependencies, an unfiltered scroll list is a bad experience. Group by
   316‚Üí  Initializr's own categories (Web, SQL, NoSQL, Security, Messaging, Ops, etc.) with category
   317‚Üí  headers in the list.
   318‚Üí- Show a spinner during network calls (metadata fetch, project generation download) with
   319‚Üí  elapsed time, and a clear failure state with retry guidance.
   320‚Üí- End with a colorized summary: what was generated, file tree (top 2 levels), and copy-paste
   321‚Üí  next-step commands (`cd`, build command matching build tool, run command).
   322‚Üí- Respect `NO_COLOR` env var and `--no-color` flag. Respect `CLICOLOR_FORCE`.
   323‚Üí- All user-facing strings centralized (not scattered `println!`) so the tool could add i18n
   324‚Üí  later without a rewrite ‚Äî a simple `messages.rs` module is enough for v1, no need for a
   325‚Üí  full i18n crate.

   329‚Üí## 6. Production-grade engineering requirements
   330‚Üí
   331‚ÜíThese are non-negotiable for calling this "production grade" ‚Äî treat as a checklist.
   332‚Üí
   333‚Üí### 6.1 Error handling
   334‚Üí- `thiserror` for typed library errors in `springup-core` (each failure mode is a distinct
   335‚Üí  variant: `InitializrUnreachable`, `InvalidDependency { id, suggestion }`,
   336‚Üí  `TemplateRenderError`, `IoError`, `InvalidProjectMetadata`, etc.).
   337‚Üí- `anyhow` (or `color-eyre` for prettier terminal output) at the `springup-cli` boundary for
   338‚Üí  ergonomic error propagation and nicely formatted terminal error reports with backtraces in
   339‚Üí  `--verbose` mode.
   340‚Üí- Never `unwrap()`/`panic!()` on any path reachable from user input or network I/O. `unwrap()`
   341‚Üí  is acceptable only on genuinely-infallible internal invariants, and even then prefer
   342‚Üí  `expect("reason")` documenting why it's infallible.
   343‚Üí- Every network call has a timeout (`reqwest` client configured with sane connect/read
   344‚Üí  timeouts) and a bounded retry with backoff for transient failures (`backoff` crate or
   345‚Üí  hand-rolled exponential backoff, 3 attempts max).
   346‚Üí
   347‚Üí### 6.2 Logging & diagnostics
   348‚Üí- `tracing` + `tracing-subscriber` for structured logging. `--verbose`/`-v` (repeatable, `-vv`
   349‚Üí  for debug, `-vvv` for trace) controls log level. Default: only user-facing progress output,
   350‚Üí  no log noise.
   351‚Üí- `--dry-run` and `--verbose` should make it possible to fully understand what the tool *would
   352‚Üí  do* without needing to read source.
   353‚Üí
   354‚Üí### 6.3 Config management (global user config)
   355‚Üí- Location via the `directories` crate (`ProjectDirs::from("dev", "springup", "springup")`)
   356‚Üí  ‚Üí e.g. `~/.config/springup/config.toml` on Linux, correct platform equivalents on
   357‚Üí  macOS/Windows.
   358‚Üí- Stores: default group-id, default author/org for READMEs, default Java version, default
   359‚Üí  build tool, preferred package manager mirror (if ever needed), telemetry opt-in flag
   360‚Üí  (default OFF ‚Äî see ¬ß6.7), color preference.
   361‚Üí- `springup config set group-id dev.raghavarora`, `springup config get group-id`,
   362‚Üí  `springup config list` ‚Äî implement properly, not just a TODO.
   363‚Üí- Precedence, most specific wins: CLI flags > project-local `.springuprc.toml` (if present in
   364‚Üí  cwd) > global user config > hardcoded defaults. Document this precedence in `--help` and
   365‚Üí  README.
   366‚Üí
   367‚Üí### 6.4 Testing
   368‚Üí- Unit tests in `springup-core` for: `ProjectPlan` validation logic, template context
   369‚Üí  derivation, dependency-id fuzzy matching, manifest serialization round-trip.
   370‚Üí- Integration tests in `tests/` that actually run the compiled binary (`assert_cmd` crate)
   371‚Üí  against a **mocked** Initializr server (`wiremock` or `mockito` crate) ‚Äî never hit the real
   372‚Üí  network in CI. Assert on generated file tree structure and key file contents (e.g. Dockerfile
   373‚Üí  contains the right base image tag for the chosen Java version).
   374‚Üí- Golden-file / snapshot tests (`insta` crate) for generated file contents (Dockerfile,
   375‚Üí  GitHub Actions YAML, README) so template regressions are caught immediately on any template
   376‚Üí  edit.
   377‚Üí- A real (non-mocked) opt-in "smoke test" behind a feature flag or separate CI job that hits
   378‚Üí  the live Initializr API once, to catch upstream API drift ‚Äî run on a schedule (e.g. nightly
   379‚Üí  GitHub Actions cron), not on every PR.
   380‚Üí
   381‚Üí### 6.5 CI/CD for the tool itself
   382‚Üí- GitHub Actions workflow(s):
   383‚Üí  - `ci.yml`: on every PR/push ‚Äî `cargo fmt --check`, `cargo clippy -- -D warnings`,
   384‚Üí    `cargo test --all-features`, run on a matrix of `ubuntu-latest`, `macos-latest`,
   385‚Üí    `windows-latest`.
   386‚Üí  - `release.yml`: on tag push (`v*`) ‚Äî build release binaries for at minimum:
   387‚Üí    `x86_64-unknown-linux-gnu`, `x86_64-unknown-linux-musl` (static, for broad Linux compat),
   388‚Üí    `aarch64-unknown-linux-gnu`, `x86_64-apple-darwin`, `aarch64-apple-darwin`,
   389‚Üí    `x86_64-pc-windows-msvc`. Use `cargo-dist` (recommended ‚Äî it generates the release
   390‚Üí    workflow, install scripts, and Homebrew/Scoop manifests for you) rather than hand-rolling
   391‚Üí    cross-compilation matrix YAML from scratch.
   392‚Üí- `cargo-dist` also generates the `curl | sh` install script and can publish a Homebrew tap
   393‚Üí  and Scoop manifest automatically ‚Äî use it rather than reinventing this.
   394‚Üí- Also publish to crates.io (`cargo publish`) as a secondary install path for Rust users who
   395‚Üí  prefer `cargo install springup`.
   396‚Üí
   397‚Üí### 6.6 Documentation
   398‚Üí- Top-level `README.md`: install instructions for every distribution channel, quickstart GIF
   399‚Üí  or asciinema-style terminal recording reference, full command reference, architecture
   400‚Üí  overview linking to `ARCHITECTURE.md`.
   401‚Üí- `ARCHITECTURE.md`: the module layout from ¬ß3.2, the `ProjectPlan` data flow, how to add a
   402‚Üí  new "extra" template.
   403‚Üí- `CONTRIBUTING.md`: how to run tests, how to add a new architecture skeleton or extra, PR
   404‚Üí  checklist (fmt/clippy/tests must pass).
   405‚Üí- Every public function/struct in `springup-core` has doc comments (`cargo doc` should
   406‚Üí  produce something genuinely useful, not empty stubs).
   407‚Üí
   408‚Üí### 6.7 Privacy / telemetry
   409‚Üí- If any telemetry is ever added (usage analytics), it must be **opt-in**, off by default,
   410‚Üí  clearly disclosed in README, and disableable via `springup config set telemetry false` and
   411‚Üí  an env var (`SPRINGUP_TELEMETRY=0`). For v1, ship with **no telemetry at all** ‚Äî simplest
   412‚Üí  and most trustworthy default; mention this explicitly in README as a stated principle.
   413‚Üí
   414‚Üí### 6.8 Security
   415‚Üí- Never execute arbitrary code from downloaded templates (the Initializr zip is data, not
   416‚Üí  code ‚Äî treat it as such; don't `eval`/shell-exec anything extracted from it).
   417‚Üí- Validate zip extraction against path traversal (zip-slip vulnerability) ‚Äî use a
   418‚Üí  well-maintained zip crate (`zip` crate) and explicitly sanitize/verify every extracted
   419‚Üí  path stays within the target directory before writing.
   420‚Üí- Pin dependency versions in `Cargo.lock`, commit it (this is a binary, not a library others
   421‚Üí  depend on ‚Äî lockfile should always be committed).
   422‚Üí- Run `cargo audit` in CI as a scheduled job to catch known vulnerabilities in dependencies.
   423‚Üí
   424‚Üí### 6.9 Performance
   425‚Üí- Cold start (no cache) to first wizard prompt should feel instant (<100ms) ‚Äî metadata fetch
   426‚Üí  happens *after* the prompts that don't need it (project name, output dir) start, not
   427‚Üí  before, so perceived latency is hidden. Fetch metadata concurrently with the first 1-2
   428‚Üí  wizard prompts using an async task, `await` it only when the wizard reaches the
   429‚Üí  dependency-selection step.
   430‚Üí- Use `tokio` (async runtime, since we're doing concurrent network + prompt work) with a
   431‚Üí  minimal feature set (`rt-multi-thread`, `macros`, `time` ‚Äî not the full `full` feature
   432‚Üí  flag, to keep binary size and compile time down).
   433‚Üí- Release profile: enable LTO and strip symbols in `Cargo.toml` `[profile.release]`
   434‚Üí  (`lto = true`, `strip = true`, `codegen-units = 1`) for smaller, faster binaries ‚Äî the
   435‚Üí  build-time cost is worth it for a tool distributed as prebuilt binaries.
   436‚Üí

   439‚Üí## 7. Key crate choices (recommended, justify any deviation)
   440‚Üí
   441‚Üí| Concern | Crate |

   447‚Üí| HTTP client | `reqwest` (rustls-tls feature, not native-tls, for easier musl static builds) |
   448‚Üí| Templating | `minijinja` |
   449‚Üí| Embedded assets | `rust-embed` |
   450‚Üí| Zip handling | `zip` |
   451‚Üí| Serialization | `serde` + `serde_json` + `toml` |
   452‚Üí| Error types (lib) | `thiserror` |
   453‚Üí| Error handling (bin) | `color-eyre` |
   454‚Üí| Logging | `tracing` + `tracing-subscriber` |
   455‚Üí| Config dirs | `directories` |
   456‚Üí| Fuzzy matching | `strsim` |
   457‚Üí| Colored output | `owo-colors` or `console` (pick one, `dialoguer` already pulls in `console`) |
   458‚Üí| Testing (CLI) | `assert_cmd` + `predicates` |
   459‚Üí| Testing (mocking) | `wiremock` |
   460‚Üí| Testing (snapshots) | `insta` |

   465‚Üí## 8. Milestone plan (build in this order)
   466‚Üí
   467‚Üí1. **Skeleton**: workspace with the three crates, `clap` CLI parsing `new` command with all
   468‚Üí   flags (no logic yet ‚Äî just parse and print the resolved struct). CI running fmt/clippy/test
   469‚Üí   on empty test suite.
   470‚Üí2. **Initializr client**: metadata fetch + cache, `starter.zip` fetch + extraction to temp
   471‚Üí   dir, with mocked-server integration tests.
   472‚Üí3. **Non-interactive `new`**: full flag-driven path works end-to-end ‚Äî generates a real,
   473‚Üí   buildable Spring Boot project with zero prompts, zero custom extras yet.
   474‚Üí4. **Custom template layer**: architecture skeletons + extras, `minijinja` rendering,
   475‚Üí   `springup.toml` manifest writing. Snapshot tests for every template.
   476‚Üí5. **Interactive wizard**: full `dialoguer` flow wrapping the same `ProjectPlan` builder,
   477‚Üí   with searchable dependency multi-select.
   478‚Üí6. **Config system**: global config file, precedence resolution, `config` subcommand.
   479‚Üí7. **Polish pass**: colorized summary output, `--dry-run`, shell completions, `--verbose`
   480‚Üí   logging, error message quality pass (every error should suggest a fix).
   481‚Üí8. **Release infrastructure**: `cargo-dist` setup, crates.io publish, README/docs finalized.
   482‚Üí9. **v1.0.0 tag.**
   483‚Üí
   484‚ÜíStop and produce a working, tested artifact at the end of *every* milestone ‚Äî don't build all
   485‚Üímodules half-finished in parallel. Each milestone should be independently `cargo test`-green
   486‚Üíand independently demoable.
